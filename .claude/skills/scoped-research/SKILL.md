# Scoped Research

특정 주제/기술/도메인에 대해, **"먼저 범위를 좁게 잡고 → 여러 축으로 훑고 → 종합"** 하는 리서치 엔진.
목표는 링크 나열이 아니라 **사용자의 실제 질문에 답하는, 근거가 붙은 의사결정용 리포트**.

## 언제 쓰나 (트리거)

- "~하려면 뭘 쓰면 좋을까 / 어떻게 접근하지" 같은 **기술 선택·설계 판단**
- "요즘 이 분야 / 이 업계 어떻게 해?" 같은 **트렌드·표준·베스트프랙티스**
- "선두 기업/서비스는 뭘 쓰나" 같은 **레퍼런스 스캐닝**
- "이 두 개(A vs B) 중에 / 이걸 붙이려면" 같은 **비교·통합 조사**
- 사용자가 질문 끝에 스킬을 명시적으로 붙이는 경우

단발성 사실 확인(환율, 특정 인물의 현재 직책 등 검색 한 번이면 끝)은 이 스킬 대상이 아니다.

## 알고리즘 (이 순서를 지킨다)

### STEP 1 — 요청 해부 (Parse)

요청에서 다음을 추출한다. 추측 금지, 요청에 근거해서 뽑는다.

- **Entities**: 등장한 제품·기술·조직·표준·개념 (예: `Gemini 3.5 Live Translate`, `SimulWhisper`, `speaker diarization`)
- **Intent**: 사용자가 진짜 원하는 것 = 동사로 정리 (예: "전사 결과에 화자분리를 붙여 서비스화하는 방법 결정")
- **Request type**: 아래 중 하나로 분류
  - `기술선택/설계` · `비교평가` · `트렌드/표준 파악` · `레퍼런스 스캐닝(선두기업)` · `통합/구현 방법` · `시장/BM 조사` · `학습/입문 로드맵`
- **Constraints**: 언어·지역(국내/글로벌)·실시간성·비용·규모·규제 등 요청에 드러난 제약

이 단계 결과를 리포트 맨 앞 "요청 해석"에 그대로 싣는다.

### STEP 2 — 분야 설정 (Scope) — **이 스킬의 핵심**

**전략: 가능한 한 가장 좁게 잡는다. 좁게 잡을 수 없거나 결과가 빈약할 때만 한 단계씩 위로 확장한다.**

범위는 3단 계층으로 생각한다:

```
L1 (가장 좁게)  : 구체 태스크/서브도메인   예) 실시간 화자분리(streaming speaker diarization)
L2 (중간)       : 상위 도메인             예) Speech AI / 음성 ASR
L3 (넓게)       : 산업/대분류             예) AI·ML 일반
```

- **항상 L1부터 시작**한다. L1이 검색으로 잘 잡히고 자료가 충분하면 거기서 리서치한다.
- L1이 너무 니치해서 (a) 검색 결과가 거의 없거나 (b) 정의 자체가 모호하면 → **L2로 한 단계 확장**. 그래도 부족하면 L3.
- 확장했더라도 **L1 관점의 질문에 답하는 걸 잊지 않는다.** (넓힌 건 재료를 더 구하려는 것이지 질문을 흐리려는 게 아니다.)
- 요청에 여러 도메인이 섞이면(예: 음성AI + 서비스기획) **주(主) 도메인 1개 + 부(副) 도메인**으로 나눠 잡는다.

리포트에 "설정한 분야: L1 = …, (필요시 확장) L2 = …, 이유: …"를 명시한다.

### STEP 3 — 리서치 스윕 (Sweep)

STEP 2에서 잡은 분야를 기준으로 아래 **축(axis)** 들을 훑는다.
축마다 검색어를 분야 용어로 구체화한다. (검색어는 짧고 구체적으로, 1~6단어.)

기본 순서:

1. **Global standard / trend** — 그 분야의 국제 표준·정석·최신 흐름
2. **Domestic standard / trend** — 국내(한국) 표준·사례·관행 (한국어로 검색)
3. **선두 기업/조직 → 그들의 스택** — 이 분야 리더 조직을 먼저 찾고 → 그 조직이 **우선 쓰는 기술/프레임워크/BM/방식/workflow**를 판다
   - **가능하면 항상 3개까지 확장**한다. 1개만 보고 끝내지 말 것.
   - 3개를 **같은 기준으로 비교**(채택 기술 / 아키텍처·워크플로우 / BM / 강점·차이)한 뒤 **공통점(=사실상의 업계 관행)과 차이점(=선택지)** 을 종합한다.
   - 리더가 3개 미만인 진짜 니치면 있는 만큼만 하고 "이 분야는 리더가 N개뿐"이라고 명시한다.
4. **Community** — 개발/교육/아티클 커뮤니티의 실전 논의 (HN, Reddit, dev.to, 국내 기술블로그 등)
5. **News** — 최근 발표·릴리스·동향
6. **GitHub** — 관련 OSS 중 stars 많은/뜨는 저장소, 사실상의 표준 라이브러리
7. **General web** — 위에서 안 잡힌 갭 메우기 (google web search)
8. **Official docs — 우선순위 최상 (1차 출처)** — anthropic / openai / microsoft / google / amazon / meta / apple 등 벤더 공식 문서 (해당되는 것만)
   - 공식 문서는 **가장 권위 있는 1차 출처**로 취급한다. 스윕 순서상 뒤에 있지만 **관련 벤더가 있으면 반드시 확인**하고, 검색 결과의 공식 URL은 `web_fetch`로 본문을 읽는다.
   - 어떤 축(블로그·커뮤니티·뉴스)과 공식 문서가 **상충하면 공식 문서를 우선**한다. 종합·검증 단계에서 사실 판단의 기준선은 공식 문서다.

**축은 고정된 의무가 아니라 intent에 따라 가중치를 준다:**
- `기술선택/설계·통합/구현` → GitHub, Official docs, Community, 선두기업 스택에 무게
- `트렌드/표준 파악` → Global/Domestic trend, News에 무게
- `레퍼런스 스캐닝(선두기업)` → 선두 기업 스택, Community에 무게
- `시장/BM 조사` → 선두 기업 BM, News, Domestic trend에 무게

**단, Official docs는 intent와 무관하게 항상 높은 우선순위다.** 해당 벤더가 등장하면 어떤 요청이든 공식 문서를 확인한다.

관련 없는 축은 건너뛴다. 한 번의 검색이 여러 축을 커버하면 중복 검색하지 않는다.

**검색 예산 (중요):**
- 복잡도에 맞춰 스케일: 보통 8~20회. 각 축을 서로 다른 검색어로.
- 스니펫만으로 부족하면 핵심 소스는 `web_fetch`로 본문을 읽는다.
- 필요 검색이 30회를 크게 넘길 정도로 광범위하면, 사용자에게 "범위를 좁히거나 Research 기능을 쓰는 게 낫다"고 제안한다.
- 상세한 소스 지도(어느 축에서 어디를 뒤질지, 국내 소스 목록 포함)는 `references/source-playbook.md`를 참고.

**품질 규칙:**
- 놀라운 결과라도 대체로 신뢰하되, 정치·유사과학·논쟁 영역은 회의적으로 교차검증한다.
- 축마다 상충하는 정보가 있으면 그대로 드러낸다. 지어내지 않는다. 못 찾으면 "못 찾음"이라고 쓴다.
- 저작권: 원문 인용은 최소화(15단어 미만, 소스당 1회 이하), 대부분 자기 문장으로 요약. 가사·시 등은 재현 금지.
- **출처 추적(필수):** 조사하면서 각 주장에 쓴 소스를 그때그때 기록한다. 소스마다 `제목 · URL · 어느 축 · 무슨 주장을 뒷받침하는지`를 남긴다. **실제로 열어본(검색결과에 뜬/`web_fetch`한) URL만** 기록한다 — URL을 지어내거나 기억으로 쓰지 않는다. 이 기록이 리포트의 출처 섹션과 report-writer 핸드오프의 재료가 된다.

### STEP 4 — 종합 (Synthesize)

수집한 걸 축별로 나열만 하지 말고, **STEP 1의 intent에 답하는 형태로** 엮는다.
- 실제 질문(예: "화자분리를 붙이려면?")에 대한 **직접적인 답/권장안**을 먼저 낸다.
- 권장안의 **근거를 리서치 축과 연결**하고, **각 핵심 주장에 출처를 매핑**한다 (예: "선두기업 X가 이 방식[출처] → GitHub의 Y가 사실상 표준[출처] → 공식문서 Z에서 지원[출처]"). 근거 없는 단정은 넣지 않는다.
- **선두 3사 비교 종합**: 3개 조직에서 공통으로 보이는 것 = 업계 관행(안전한 기본값), 갈리는 것 = 사용자가 골라야 할 선택지 — 이렇게 정리해 권장안에 반영한다.
- 트레이드오프·대안·리스크를 정리한다.
- 공식 문서와 다른 소스가 충돌하면 공식 문서를 기준으로 삼는다.
- Claude는 자문사가 아니므로 투자·법률성 판단은 단정하지 않는다.

### STEP 5 — 리포트 작성 (Report)

아래 템플릿으로 리포트를 작성한다(아직 최종 전달 전 — STEP 6 검증을 통과해야 내보낸다).
**인라인(대화)로 보여주되**, 사용자가 파일/문서를 원하면 문서화 스킬(report-writer, docx 등)로 넘긴다.
출처 섹션은 report-writer가 그대로 받아 쓸 수 있게 **구조화**한다(제목·URL·축·뒷받침 주장). 본문 각 주장에는 `[S1]`처럼 출처 번호를 달아 매핑한다.

### STEP 6 — 검증 게이트 (Verify) — **통과 못 하면 리포트 내보내지 않는다**

리포트를 사용자에게 내보내기 **전에**, 스스로 verifier 역할로 아래 체크리스트를 냉정하게 돌린다.
"내가 작성한 것"이라는 편향을 버리고, 처음 보는 결과물을 감사(audit)하듯 본다.

체크 항목:
1. **Intent 적합성** — 리포트가 STEP 1의 실제 질문에 답하는가? 엉뚱한 주제로 샜는가?
2. **스코프 일치** — STEP 2에서 잡은 분야 안에 있는가? 다른 분야 자료를 억지로 붙였는가?
3. **주장-출처 매핑** — 핵심 주장마다 `[S#]`로 실제 소스가 붙어 있는가? 출처 없는 단정이 있는가?
4. **출처 진위** — 인용한 URL/조직/수치가 **실제로 검색·fetch로 확인한 것**인가? 기억이나 추측으로 지어낸 것은 없는가? (환각 소스 = 즉시 실패)
5. **공식 문서 반영** — 관련 벤더가 있으면 공식 문서를 확인·반영했는가? 공식문서와 충돌하는 주장을 그대로 두지 않았는가?
6. **선두 3사** — 가능한데도 1개만 보고 끝내지 않았는가? (니치라 불가하면 그 사실을 명시했는가?)
7. **상충/공백 표기** — 상충 정보와 "못 찾음"을 숨기지 않고 드러냈는가?

**판정:**
- 항목 하나라도 미달이면 → **해당 부분을 재검색/수정한 뒤 다시 검증**한다. 통과할 때까지 반복.
- 특히 4번(환각 소스)이나 1·2번(intent/스코프 이탈)이 걸리면 리포트를 절대 그대로 내보내지 않는다.
- 검증을 통과하면 리포트 맨 끝 "검증" 섹션에 통과 사실과 (있었다면) 무엇을 보강했는지 한 줄로 남긴다.
- 광범위해서 도저히 확신이 안 서면, 억지로 채우지 말고 "이 부분은 불확실 — 범위를 좁히거나 추가 조사 필요"라고 사용자에게 정직하게 알린다.

```
- … (본문의 [S#]와 1:1 대응. 실제로 열어본 URL만.)

## 검증 (Verification)
- ✅ 통과: intent 응답 / 스코프 일치 / 주장-출처 매핑 / 공식문서 반영 / 선두 3사 확인
- (미달 항목이 있으면 무엇을 보강했는지 한 줄)
```

---

## 실행 예시 (요약)

**입력:** "Gemini 3.5 Live Translate나 SimulWhisper로 전사할 때 화자분리 붙여서 서비스하려면?"

- STEP1 → entities: Gemini Live Translate, SimulWhisper, speaker diarization / intent: 스트리밍 전사에 화자분리를 얹어 서비스화 / type: 통합·구현 방법
- STEP2 → L1 = 실시간(streaming) 화자분리 + streaming ASR 통합. 니치하지만 자료 있음 → L1 유지, 부도메인 = 실시간 서비스 아키텍처
- STEP3 → 공식문서(각 ASR 벤더) 우선 확인 + GitHub(pyannote·diart 등) + 선두기업 **3개**(예: AssemblyAI·NVIDIA Riva·Amazon/Azure/Google 중 3사) 스택 비교 + 커뮤니티(실시간 diarization 지연 이슈) + 트렌드
- STEP4 → 3사 공통점(=관행)/차이점(=선택지) 종합 → "스트리밍은 화자경계가 유동적이라 이런 파이프라인 권장 …" 질문에 답 + 각 주장에 [S#] 출처
- STEP5 → 리포트 출력(구조화 출처 포함), 원하면 report-writer로 문서화
- STEP6 → intent/스코프/출처진위/공식문서/3사 검증 통과 확인 후 전달

---

## 주의

- **범위를 좁히는 게 항상 먼저**다. 넓게 시작하면 리포트가 물에 물 탄 듯 뻔해진다.
- 축을 기계적으로 다 채우지 말고 intent에 맞게 취사선택·가중. 단 **공식 문서는 항상 우선**, **선두 기업은 가능하면 3개**.
- 링크 덤프가 아니라 **답을 내는 문서**를 만든다. 모든 핵심 주장에 출처를 매핑.
- **STEP 6 검증 게이트를 건너뛰지 않는다.** 특히 지어낸 출처·intent 이탈은 통과 불가.
- 확실치 않으면 "못 찾음/불확실"을 명시하고, 검색을 더 하거나 사용자에게 범위 조정을 제안한다.
